# РостЗнаний: Stage 1 plan

## Текущая архитектура

- Статический лендинг: `index.html`, `styles.css`, `script.js`.
- Локальные изображения находятся в `assets/`.
- Production build запускается через `npm run build` и формирует `dist/client`.
- Данные преподавателей, тарифов, историй и формы сейчас частично зашиты в HTML.
- Интерактивность главной страницы включает FAQ, раскрытие отзывов, выбор тарифа и демо-форму пробного урока.
- Публикация рассчитана на GitHub Pages по repository base path, поэтому новые переходы должны быть относительными.

## Сильные стороны

- Уже сформированная визуальная идентичность и полный маркетинговый сценарий лендинга.
- Локальные качественные изображения учеников, преподавателей и вузов.
- Существующая доступная форма заявки и понятные точки конверсии.
- Простой стек без зависимости от backend и внешних сервисов.

## Слабые стороны

- Нет единого источника данных и конфигурации экзаменационного сезона.
- Пользователь не может оценить разрыв между текущим уровнем и целью.
- Тариф, преподаватель и время урока не связаны с учебной задачей пользователя.
- Демо-заявка не сохраняет контекст выбора и не имеет локального адаптера.
- В истории ОГЭ используется неподходящая стобалльная шкала.

## Существующие пользовательские сценарии

- Просмотр направлений, процесса обучения, преподавателей, отзывов и тарифов.
- Раскрытие подробностей о преподавателе и дополнительных отзывов.
- Выбор тарифа с переходом к заявке.
- Отправка нижней формы или многошаговой демо-формы пробного урока.

## План реализации

1. Создать единый JS-конфиг сезона и структурированный источник данных со стабильными id.
2. Исправить отображение результата ОГЭ и добавить ненавязчивую подпись portfolio concept.
3. Добавить безопасные относительные CTA на маршрут `trajectory/`.
4. Реализовать профиль пользователя из восьми шагов с черновиком в versioned localStorage.
5. Добавить по 12 оригинальных вопросов для пяти предметов и детерминированный выбор 8 вопросов.
6. Реализовать deterministic trajectory engine с отдельной логикой ЕГЭ и ОГЭ.
7. Сформировать экран результата, визуальную траекторию и учебный план на 8–16 недель.
8. Связать результат с существующими преподавателями, тарифами и будущими слотами.
9. Использовать единый booking flow и локальный booking adapter с максимум пятью демо-заявками.
10. Обновить build script, выполнить build и локальную браузерную проверку.

## Визуальное направление

- Палитра trajectory: `#07152f` (navy), `#0d2450` (surface), `#f5f8ff` (text), `#ff9f2f` (orange action), `#59c8ff` (data accent), `#8ce6c0` (positive state).
- Типографика остаётся системной и совместимой с текущим лендингом; крупные числа и краткие utility-labels используются только для данных результата.
- Основная композиция: спокойная тёмная рабочая поверхность, ясный stepper, широкая карточка текущего шага и боковое резюме на desktop.
- Запоминающийся элемент: линия «Траектории балла», которая постепенно заполняется от текущего уровня к цели и служит навигационным мотивом, а не декором.
- Анимация ограничена сменой шагов и заполнением прогресса; при `prefers-reduced-motion` отключается.

## Критерии готовности

- Главная страница и `/trajectory/` собираются в `dist/client`.
- Одинаковые входные данные и ответы дают одинаковый результат.
- ЕГЭ отображается в баллах, ОГЭ только в отметках и уровнях готовности.
- Пользователь проходит профиль, 8 вопросов, получает план, выбирает преподавателя, тариф и слот, затем оформляет одну демо-заявку.
- Все данные и localStorage имеют стабильные id/версии и безопасные fallback.
- Нет абсолютных ссылок, ведущих мимо repository base path, localhost-ссылок и горизонтального overflow на целевых viewport.
- Build завершается без ошибок, а локальный checkpoint содержит только файлы «Рост знаний».
